# No3d Asset Developer v2 — Plan

> Renamed from "NO3D Tools Asset Utility" to distinguish from the Obsidian-based No3d Manager.
> This addon lives in Blender. It's the creation/export/documentation tool.
> No3d Manager (Obsidian) is the viewing/organizing/syncing tool.

## Design Principle: Each Tool Writes What It Knows

The addon knows Blender data — asset name, type, blender version, tags from asset metadata, export date, source file. It writes this as frontmatter.

The addon does NOT know — Cloudinary URLs, carousel order, thumbnail selection, pricing, hosted media state. That's downstream. The Obsidian plugin (or a sync CLI) generates the product JSON from frontmatter + folder state when it has the full picture.

```
Blender (No3d Asset Developer)
  → desc_{Name}.md     (frontmatter + blank description)
  → {Name}.blend       (asset file)
  → icon_{Name}.png    (thumbnail)
  → notes_{Name}.md    (dev notes, if any)
                ↓
Obsidian (No3d Manager) or sync CLI
  → reads frontmatter + scans folder + media state
  → generates {Name}.json (complete product JSON)
                ↓
sync-product-to-supabase.js
  → reads JSON → pushes to Supabase
```

**No JSON is generated by the addon.** If an old `{Name}.json` exists from v1, the addon leaves it alone — it's the downstream tool's responsibility now.

## New Feature: Quick Dev Notes

A fast note-taking system inside Blender for jotting down rough ideas, TODOs, and instructions during asset development. Notes are exported with the asset and intended to be read by a scheduled Claude agent.

**How it works:**
- Text field in the N-panel sidebar (NO3D tab)
- Each note is timestamped and tagged to the active asset (or global if no asset selected)
- Notes accumulate in a per-asset list during a session
- On export: notes are written to `notes_{AssetName}.md` in the product folder
- A scheduled Claude agent sweeps these files and acts on them

**Note format (exported):**
```markdown
---
asset: "Dojo Bool v4.5"
created: 2026-03-26
source: blender
status: unprocessed
---

## Dev Notes

- [14:23] Need to add a fallback for when the input mesh has no faces
- [14:45] The boolean merge mode should default to "Union" not "Intersect"
- [15:10] Screenshot the modifier panel for docs
- [15:30] This would pair well with the Array Scatter tool — mention in cross-references
```

**UI:**
- Sidebar panel: text input + "Add Note" button
- Notes list below with timestamps
- "Clear Notes" button
- Notes persist in `window_manager` properties during session (not saved to .blend)
- Exported on asset export (alongside .blend, frontmatter, thumbnail)

### Frontmatter Schema

The frontmatter schema is shared with the Obsidian plugin — both tools read and write the same fields. The addon writes what it knows at export time:

```yaml
---
title: "Dojo Support Bracket"
handle: "dojo-support-bracket"
price: "0.00"
sku: "NO3D-TOOLS-DOJO-SUPPORT-BRACKET"
status: draft
vendor: "The Well Tarot"
product_type: "Blender Add-on"
tags:
  - blender
  - geometry-nodes
metafields:
  asset_type: "OBJECT"
  blender_version: "5.1+"
  export_date: "2026-03-26"
  source_file: "Dojo Bracket Generator.blend"
  blend_file: "Dojo Bracket.blend"
  thumbnail: "icon_Dojo Bracket.png"
changelog:
  - version: "1.0.0"
    date: "2026-03-26"
    description: "Initial export"
---

# Dojo Support Bracket

<!-- Description to be written in Obsidian -->
```

**What the addon populates:**
- `title` — from asset name
- `handle` — auto-slugified from title
- `price` — defaults to `"0.00"` (set later in Obsidian)
- `sku` — generated from title pattern
- `status` — always `draft` on initial export
- `vendor` — defaults to `"The Well Tarot"`
- `product_type` — `"Blender Add-on"` (configurable in addon preferences)
- `tags` — from asset's Blender tags + auto-detected type tags (e.g., `geometry-nodes` if GeoNode asset)
- `metafields` — asset_type, blender_version, export_date, source_file, blend_file, thumbnail
- `changelog` — initial entry

**What the addon does NOT populate** (downstream responsibility):
- `hosted_media` — added after Cloudinary upload
- `carousel_media` — set in Obsidian plugin media manager
- `thumbnail_image` — set in Obsidian plugin media manager
- `main_image` — set in Obsidian plugin media manager
- `excluded_carousel_media` — set in Obsidian plugin media manager

### Backward Compatibility: Existing v1 Products

When exporting to a folder that already has a `{Name}.json` from v1:
- The addon creates `desc_{Name}.md` with frontmatter populated from the existing JSON (reads title, handle, price, tags, status, vendor, etc. from the JSON to seed the frontmatter)
- The old JSON is left in place — the downstream tools decide when to take over JSON generation
- If a `desc_{Name}.md` already exists, the addon updates its frontmatter fields but preserves the markdown body (description content)

## .blend Export: Template-Based Approach (v2)

### Why v1 Breaks

v1 uses `bpy.data.libraries.write()` to write individual data blocks to a new .blend. This causes three problems:

1. **No scene/unit settings.** `libraries.write()` creates a "library" .blend with no Scene object. When opened, Blender applies default settings — wrong units, wrong scale, wrong color management. Every exported asset has different settings depending on who opens it first.

2. **Empty or missing files.** The dependency isolation logic (filtering shared materials, removing asset-marked node groups) sometimes over-filters, writing an empty data block set. Result: a .blend file that contains nothing.

3. **Smuggled assets.** `libraries.write()` automatically pulls in transitive dependencies. If a material is shared with another asset, or a node group references another node group that's marked as an asset, those unwanted assets leak into the export. The v1 code tries to filter them, but the filtering is fragile.

### v2 Solution: Template + Subprocess Append

The addon ships with a `_export_template.blend` — a minimal file with the correct scene and unit settings, empty of all content.

For each asset export, the addon:

1. **Spawns a background Blender subprocess** that opens the template
2. **Appends the target asset** from the source file using `bpy.ops.wm.append()`
3. **Runs a cleanup pass** that strips any smuggled assets (data blocks marked as assets that aren't the target)
4. **Saves** the result to the output path

```
Current Session (untouched)          Background Subprocess
                                     ┌──────────────────────────┐
                                     │ Opens _export_template.blend │
[User's .blend file]  ──append──→    │ Appends target asset     │
                                     │ Strips smuggled assets   │
                                     │ Saves to output path     │
                                     └──────────────────────────┘
                                              ↓
                                     AssetName/AssetName.blend
                                     (correct units, clean, small)
```

### Template Settings

The `_export_template.blend` contains:

```
Scene Settings:
  Units: Metric
  Unit Scale: 0.001
  Length: Millimeters

Render Settings:
  Engine: Cycles (or EEVEE — whichever is standard)
  Color Management: Standard/sRGB

World: Default gray (or none)
Objects: None
Materials: None
Collections: Default "Collection" only
```

This file is bundled with the addon in the addon directory. It's created once and never modified.

### Cleanup Pass: Preventing Smuggled Assets

After appending, the subprocess runs a cleanup that:

1. Identifies the target asset by name
2. Scans ALL data blocks in the file for `asset_data is not None`
3. Any data block that is marked as an asset AND is not the target → `asset_data.clear()` (unmarks it as an asset but keeps the data if it's a dependency)
4. Orphan data blocks with no users → removed via `bpy.ops.outliner.orphans_purge()`

This means: if Material B is shared between Asset A and Asset B, and you export Asset A, Material B comes through (because Asset A needs it) but its asset marking is stripped. It exists as a dependency, not as a separate asset. No duplicates in the Asset Browser.

### Overwrite Behavior (Retained from v1)

Exporting to a folder that already contains `AssetName.blend` overwrites it cleanly. No `.001` suffixes, no duplicates. The subprocess writes directly to the target path.

### Performance

Spawning a Blender subprocess per asset adds ~2-3 seconds overhead (template load + append + save). For your catalog:
- Single asset export: ~3 seconds (imperceptible in workflow)
- Full catalog re-export (50 assets): ~2-3 minutes
- Thumbnail-only export: no subprocess needed (runs in current session)

### Export Script (runs in subprocess)

The addon bundles a `_export_single_asset.py` script that the subprocess executes:

```
blender --background _export_template.blend --python _export_single_asset.py -- \
  --source "/path/to/source.blend" \
  --asset "Dojo Support Bracket" \
  --asset-type "Object" \
  --output "/path/to/library/Dojo Bracket/Dojo Support Bracket.blend"
```

## Future Explorations (Not v2)

### Node Group Introspection & Auto-Documentation

Validated in proof-of-concept (headless Blender introspection of Dojo Bracket Generator, March 2026). The Blender Python API provides full access to node trees:

- `node_tree.nodes` — all nodes with types, properties, socket connections
- `node_tree.links` — complete connection graph
- `node_tree.interface` — group inputs/outputs with types, defaults, ranges
- `bl_rna.properties` — generic discovery of node-specific settings (Math operation, Boolean mode, etc.)

**The approach:** Serialize a node group to structured JSON, generate a semantic summary (interface table, data flow spine, branch paths, node inventory), feed to an LLM for documentation generation.

**Status:** API works, serialization is feasible, LLM interpretation is promising but needs iteration on the summary format. The Dojo Bracket node group (155 nodes, 18 subgroups) was successfully introspected.

**When to revisit:** After v2 is stable and the Obsidian plugin's JSON generation pipeline is working. The introspection output (`_nodetree.json`, `_nodetree.md`) would be additional export artifacts alongside the frontmatter and .blend file.
